Go Error Handling

Handling Errors the Go Way

What is an Error?

In Go, errors are values. Functions usually return an error as the last return value to indicate something went wrong.

The error Interface

type error interface { Error() string }

Basic Error Handling

You check errors using an if statement.

result, err := strconv.Atoi("123") if err != nil { fmt.Println("Error:", err) } else { fmt.Println("Result:", result) }

Creating Errors

Using errors.New

err := errors.New("something went wrong")

Using fmt.Errorf

err := fmt.Errorf("invalid value: %d", value)

Custom Errors

You can define your own error types.

type MyError struct { Message string } func (e MyError) Error() string { return e.Message }

Returning Errors from Functions

func divide(a, b int) (int, error) { if b == 0 { return 0, errors.New("division by zero") } return a / b, nil }

Handling Returned Errors

result, err := divide(10, 0) if err != nil { fmt.Println("Error:", err) return } fmt.Println("Result:", result)

Panic and Recover

Panic is used for unexpected errors. Recover can stop a panic.

func safeFunction() { defer func() { if r := recover(); r != nil { fmt.Println("Recovered:", r) } }() panic("unexpected error") }

Best Practices

  • Always handle errors.
  • Return errors instead of panicking.
  • Add context using fmt.Errorf.
  • Use panic only for unrecoverable situations.

Conclusion

Go encourages explicit and simple error handling. By treating errors as values, your programs become more reliable and easier to debug.
